A reliable, zero-API Kitchen Display System (KDS) strictly using open-source web technologies. Built to intercept emails/SMS, parse them into structured orders, and dictate kitchen flow with dynamic color timers.

How It Works: Order Capture & Parsing

Because official Swiggy/Zomato APIs are expensive and restrictive, we bypass them using standard communication channels. Here is the exact technical approach for capturing and processing orders.

Step 1: Capture

Email & SMS Interception

We rely on the platform notifications sent to the restaurant:

  • Primary (Email): The restaurant's registered Gmail/Outlook is configured with an auto-forwarding rule. All emails from "orders@swiggy.in" are instantly forwarded to a dedicated system inbox (e.g., ingest@ourkds.com).
  • Fallback (SMS Webhook): An old Android phone in the kitchen runs a free "SMS Forwarder" app. It listens for SMS from sender ID "ZOMATO". When a text arrives, the app fires an HTTP POST request containing the raw text to our Laravel `/api/webhooks/sms` endpoint.
Step 2: Parsing

Laravel Regex Engine

Once the raw text enters Laravel, it is passed through platform-specific Regular Expression (Regex) dictionaries.

// Example Swiggy Regex Pattern
preg_match('/Order ID:\s*(SWG[0-9]+)/', $text, $id);
preg_match_all('/(\d+)x\s*(.*?)(?=\n|$)/', $text, $items);

The engine scans the unstructured text, extracting the Order ID, Item list, Quantities, and Special Notes, transforming it into a structured JSON array.

Step 3: Processing

Deduplication & MySQL

Before saving, Laravel executes a critical safety check.

  • Unique Key Lock: Laravel checks MySQL for a `unique_key` (e.g., "SWIGGY_10523"). If both an SMS and an Email arrive for the same order, the second one is dropped to prevent double-cooking.
  • Commit: The clean JSON is inserted into the MySQL `orders` and `order_items` tables. The `received_at` timestamp is hard-locked.

End-to-End Operational Workflow

A step-by-step lifecycle of a single order—from the moment a customer clicks "Buy" to the moment the food leaves your kitchen.

1. The Trigger (Order Placement)

A customer places an order on Swiggy or Zomato. The delivery platform's internal system automatically generates a standard notification (Email or SMS) and sends it to the restaurant's registered contact details.

2. Silent Interception

Without any human intervention, the system captures this notification. If it's an email, an auto-forwarding rule instantly routes it to our Laravel IMAP inbox. If it's an SMS, the kitchen's fallback Android device catches the text and immediately fires a webhook to the Laravel API.

3. Extraction & DB Lock

The Laravel backend takes the raw, messy text and runs it through the Regex Engine. It extracts the exact Order ID, platform name, timestamp, and item list. It then queries the MySQL database. If the unique_key doesn't exist yet, it inserts the order with status = 'new'.

4. TV Display Broadcast

Within 5 seconds, the Alpine.js frontend (running continuously on the kitchen TV) polls the backend and detects the new record. The order instantly pops up on the screen as a large, readable card, alerting the chefs to begin prep.

5. Live Aging & Color Shifting

As the kitchen staff cooks, the timer on the order card ticks upwards. Alpine.js constantly calculates the time difference. At 0-5 minutes, the card is Green. At 6 minutes, it automatically turns Yellow. If it passes 11 minutes, it flashes Red, indicating an urgent delay.

6. Order Completion & Clearance

Once the food is packed and ready for the delivery partner, a kitchen staff member clicks the "Ready / Done" button on the dashboard using a connected mouse or touch display. This sends a quick AJAX request to Laravel, updating the MySQL status to 'completed'. The order instantly vanishes from the active TV queue, keeping the screen clean for the next rush.

Data Conversion & Logic Map

The Laravel parser outputs a strict JSON structure. The Blade frontend relies entirely on system-calculated timestamps to apply visual colors, avoiding static data traps.

  "source_type": "email_imap",
  "unique_key": "SWIGGY_SWG-891102",

{

Rule: The Laravel API returns `received_at`. Alpine.js calculates `elapsed_minutes` = `Now() - received_at` locally every second, ensuring the Green/Yellow/Red card colors shift accurately in real-time without slamming the server.
