Order Aging Display System
Lean MVP Engineering Plan

An air-tight, reliable, and cost-effective Kitchen Display System (KDS) designed specifically for cloud kitchens. Zero direct platform API dependencies, built for raw operational speed and strict reliability.

Operational Control

Eliminates missed orders and dependency on mobile apps. One central screen dictates kitchen flow.

Dynamic Aging

Automated Green/Yellow/Red visual alerts based on strict timer rules calculated from actual receipt time.

Air-tight Fallbacks

Built-in manual entry, duplicate prevention, and raw-message logging to ensure zero data loss during rushes.

Module Selected Tech Engineering Rationale (Cost & Reliability)
Capture Source Backend Email Parser (IMAP) Order emails are auto-forwarded to a dedicated inbox. A backend cron reads and parses them. 100% web-based architecture, zero mobile apps required.
Backend Framework Node.js (Express) OR Python (FastAPI) Highly abundant developer skillsets keep dev costs low. Extremely lightweight for local network hosting.
Database SQLite (Local file-based) Zero setup time, zero cloud cost. More than capable of handling 1,000+ orders/day for a single kitchen instance.
Frontend Sync HTTP Polling (Every 5 Seconds) Cheaper and faster to implement than WebSockets for MVP. Kitchens don't notice a 5-second delay.
Deduplication Composite Key: platform + order_id Air-tight logic to prevent phantom double-orders if a notification is intercepted twice.

Required Dashboard Rules

  • Oldest First: Active orders strictly sorted by oldest `received_at` timestamp.
  • Readability: Text must be legible from 6–10 feet away. High contrast.
  • Status Flow: New → Start → Ready → Done. 'Done' and 'Cancelled' vanish from active view immediately.
  • Auto-Refresh: Polling occurs without page reload. If connection drops, a visible "Offline" banner appears.

Failure Handling (Edge Cases)

  • Parse Failure: If parser breaks, save raw text to DB and display a "PARSE ERROR" blank card for manual review.
  • Missed Capture: A "Add Manual Order" floating action button (FAB) allows 10-second manual entry.
  • Massive Rush: UI must handle vertical scrolling cleanly if >12 orders are active.

Development Roadmap (4 Weeks)

Week 1: Data Capture & Parsing

Develop the IMAP email listener. Extract 50 sample real-world order emails and perfect the Regex rules to achieve >98% accuracy on test data.

Week 2: Core Backend & DB

Setup local SQLite database. Build REST endpoints for receiving parsed data, updating statuses, and manual fallback entry. Implement unique key checks.

Week 3: TV Dashboard UI

Build the frontend grid. Implement 5-second polling loop and dynamic color calculation based on timestamps. Add large status buttons.

Week 4: Stress Testing & Install

Simulate a friday night rush (50+ rapid orders). Test manual entry flow. Deploy on local Mini PC connected to kitchen TV.
